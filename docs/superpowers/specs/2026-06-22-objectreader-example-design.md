# ObjectReader — a FastMember-equivalent `IDataReader` source generator (dog-food example, design)

- **Date:** 2026-06-22
- **Status:** Approved — design. To be implemented on a new `experimental/object-reader` bookmark
  (cut from the current `experimental/matching` tip). Not on main.
- **Builds on:** Synto **Templating** (the shipped feature — `[Template]` / `[Inline]` / `Syntax<T>`)
  and the **`Synto.Diagnostics`** `[Diagnostic]` generator. Does **not** depend on Matching, though the
  reader's per-member switch construction *may* optionally exercise `Pattern.Replace`.
- **Tracking:** First deliberate "build a real, semi-complex generator with Synto" dog-food. Its primary
  by-product — a **friction log** — feeds a *separate, later* Synto-improvement spec (phased; see §11).

## 1. Motivation

[FastMember](https://github.com/mgravell/fast-member) exposes a list of POCOs as an `IDataReader`
(`ObjectReader.Create(source, members)`) so you can hand it to `SqlBulkCopy` / any ADO.NET sink — but it
does so with **runtime `Reflection.Emit` (IL)**. We want the **source-generated** equivalent: identical
call ergonomics, zero runtime reflection, the reader specialized per type at compile time.

The deeper purpose is to **dog-food Synto on a genuinely semi-complex generator** and measure *how painful
it still is to build*. Synto's whole reason to exist is to turn "a source gen would solve this" into
something you actually reach for, instead of "…but that's a ton of `SyntaxFactory` soup, so we won't."
ObjectReader is a good probe: a real, useful artifact whose generated output is a ~30-member `IDataReader`
implementation specialized per type — exactly the kind of thing that is tedious by hand and should be
where templating earns its keep. **Every place it hurts is a finding** (§11), not a failure.

A failed/abandoned WIP of this exact idea existed previously (recovered from history at `3f64b23`, removed
in `a72603b`). It was riddled with `NotImplementedException`, a non-compiling `GetValues`, `null!` template
arguments, and the deprecated hand-rolled `InterceptsLocation(file, line, char)` model. This is a
**clean redesign and rebuild**, not a revival.

## 2. Background — the surface today

- **Synto Templating.** A `[Template(typeof(Factory))]`-annotated method/type is quoted by
  `Synto.SourceGenerator` at the generator project's build time into `Factory.*()` helpers that rebuild it
  as a `SyntaxNode`. `[Inline(InlineOption.Syntax)]` parameters become syntax holes; `Syntax<T>` typed
  holes splice consumer-supplied expressions. The consumer surface is **injected `internal`**; emitted
  helpers are `file`-scoped — generated code is **self-contained, no runtime package dependency**.
- **`Synto.Diagnostics`.** `[Diagnostic("ID", title, message, category, severity, …)] partial Diagnostic
  Foo(Location?, …)` generates the `DiagnosticDescriptor` + factory. Self-contained (injects its own
  attribute). Used here for the `SOR*` diagnostics.
- **C# interceptors (net10 / Roslyn 5).** The supported model is
  `SemanticModel.GetInterceptableLocation(InvocationExpressionSyntax)` →
  `InterceptableLocation.GetInterceptsLocationAttributeSyntax()`, which emits a version-stamped,
  refactoring-safe `[InterceptsLocation(version, "data")]`. The consumer enables it per-namespace via the
  `<InterceptorsNamespaces>` MSBuild property. (The old `InterceptsLocation(filePath, line, char)` form the
  original WIP used is deprecated — we do **not** use it.)

## 3. Design decisions

- **D1 — FastMember-identical call site.** The public surface is
  `ObjectReader.Create<T>(IEnumerable<T> source, params string[] members)` → `IDataReader`, matching
  FastMember. The consumer writes nothing but the call.
- **D2 — Interceptor dispatch.** Each *constant* call site is intercepted and routed to a generated reader
  specialized to `(T, members)`. No attributes on the POCO, no registration. The exact interceptor
  binding mechanism for a generic `Create<T>` call is the **#1 implementation risk** and is validated by a
  walking-skeleton spike before any templating (§9, R1).
- **D3 — Heavy Synto (strategy A), drop to hybrid only on a wall.** The reader is generated by quoting a
  single hand-written `[Template]` *skeleton* (the invariant ~25 members) with `Syntax<>` holes for the
  per-type variable parts (`FieldCount`, the `GetValue`/`GetName`/`GetOrdinal`/`GetFieldType` switch arms).
  Where a `[Template]` genuinely cannot express a needed shape, that one piece drops to raw `SyntaxFactory`
  (hybrid) **and the drop is logged as a friction finding**. The default posture is "make Synto do it."
- **D4 — A *working* reader, not a stub.** Every `IDataReader` member is either implemented or throws
  `NotSupportedException` *with a reason*. Zero `NotImplementedException`. The functional bar is
  `DataTable.Load(reader)` and `SqlBulkCopy`-shaped consumption (see C-3).
- **D5 — Diagnose unknown members; never crash.** A member name with no matching property/field on `T`
  raises `SOR0001` (warning) and is **skipped**. A non-constant member list raises `SOR0002` and is not
  intercepted (runtime path throws, C-4). Generator exceptions become `SOR0000`, never thrown (Synto
  principle).
- **D6 — Self-contained example, co-located test.** Four projects under one
  `examples/Synto.Example.ObjectReader/` umbrella (API, Generator, Demo, Tests). The test lives *with* the
  example (it is a teaching specimen, not core-library coverage), and still joins the solution so the
  green gate runs it.

## 5. Generated output shape (the spec the snapshot pins)

For `ObjectReader.Create(people, "Name", "Age")` over `class Person { public string Name; public int Age; }`,
the generator emits one file into the consumer compilation:

```csharp
namespace Synto.Example.ObjectReader.Generated;

file sealed class ObjectReader_Person_0 : System.Data.IDataReader
{
    private readonly IEnumerator<Person> _e;
    private bool _closed;
    public ObjectReader_Person_0(IEnumerable<Person> source) => _e = source.GetEnumerator();

    // ── invariant skeleton, from ONE [Template] (identical for every T) ──
    //    Read()/Close()/Dispose()/NextResult()/Depth/IsClosed/RecordsAffected;
    //    typed getters (GetInt32/GetString/GetGuid/…) route through GetValue with a cast;
    //    indexers this[int]/this[string]; IsDBNull; GetValues(object[]);
    //    GetData/GetSchemaTable → NotSupportedException("…").

    // ── variable parts, Syntax<> holes filled per call site ──
    public int    FieldCount           => 2;
    public string GetName(int i)        => i switch { 0 => "Name", 1 => "Age", _ => throw Oor(i) };
    public int    GetOrdinal(string n)  => n switch { "Name" => 0, "Age" => 1, _ => throw NoCol(n) };
    public Type   GetFieldType(int i)   => i switch { 0 => typeof(string), 1 => typeof(int), _ => throw Oor(i) };
    public object GetValue(int i)       => i switch
    {
        0 => (object?)_e.Current.Name ?? DBNull.Value,   // null → DBNull.Value (C-3)
        1 => _e.Current.Age,
        _ => throw Oor(i),
    };
}

file static class ObjectReaderInterceptors
{
    [global::System.Runtime.CompilerServices.InterceptsLocation(1, "….")]   // from GetInterceptableLocation
    public static System.Data.IDataReader Create_0/* <T> per R1 */ (IEnumerable<Person> source, params string[] members)
        => new ObjectReader_Person_0(source);
}
```

The **interceptor signature** (generic `Create_0<T>` vs. non-generic per-type, and the cast strategy) is
left to the implementer to satisfy the C# interceptor binding rules (R1) — it is pinned by *contract*
(C-1), not prescribed here, and its awkwardness is itself friction data.

## 6. Diagnostics (`Synto.Diagnostics`, prefix `SOR`)

| ID | Title | Severity | When | Effect |
|---|---|---|---|---|
| `SOR0000` | Internal error | Error | Generator throws on a reachable path | Exception converted to a diagnostic, **never** thrown (Synto principle). |
| `SOR0001` | Member not found | **Warning** | A name in `Create(…)` has no matching public property/field on `T` | The member is **skipped**; the reader exposes only resolved members. |
| `SOR0002` | Members not constant | Warning | The member list is not compile-time constant | Call is **not** intercepted; runtime path throws (C-4). |

Severity of `SOR0001` is set to Warning per the explicit decision; revisit if "skip the column" proves too
silent (an unknown column simply vanishing may warrant Error). Captured as an open calibration note, not a
blocker.

## 7. Semantics & contracts (load-bearing)

- **C-1 — Intercepted dispatch, no reflection on the hot path.** A constant `Create<T>(…)` call binds (via
  `[InterceptsLocation]`) to a generated reader specialized to `(T, members)`. Member access in the
  generated switch arms is **direct** (`_e.Current.Name`) — no `Reflection`/`Emit`/`dynamic` at runtime.
- **C-2 — Member resolution.** `members` are matched against `T`'s **public instance properties and
  fields** by exact name. Unknown → `SOR0001` + skipped. Column ordinal = position among *resolved*
  members, in call order. (Read-only members are fine — the reader only reads.)
- **C-3 — Functionally correct `IDataReader`.** The set required by `DataTable.Load` / `SqlBulkCopy`
  (`FieldCount`, `GetName`, `GetOrdinal`, `GetFieldType`, `GetValue`, `GetValues`, `Read`, `IsDBNull`,
  `Close`/`Dispose`) is fully implemented. **Nulls surface as `DBNull.Value`** (ADO.NET expectation), and
  `IsDBNull(i)` agrees. `Read()` advances the enumerator; reading before the first `Read()` or after the
  end throws `InvalidOperationException`. Exotic members (`GetData`, `GetSchemaTable`, `GetBytes`,
  `GetChars`) throw `NotSupportedException` with a reason (matches FastMember's own behavior). **Zero
  `NotImplementedException`.**
- **C-4 — Safe un-intercepted fallback.** If a call is not intercepted (non-constant members, or
  interceptors disabled), `ObjectReader.Create` throws a descriptive `NotSupportedException` explaining the
  compile-time-constant requirement. (A runtime reflection fallback is a noted non-goal, §10.)
- **C-5 — Incremental cacheability.** No `Compilation` / `ISymbol` / `SemanticModel` / `SyntaxNode` is
  captured into cached pipeline state. The per-call-site analysis runs inside the syntax/semantic
  transform and flows out an **equatable value** (target type name + namespace, ordered resolved members
  with their type names, the interceptable-location payload, an `EquatableArray<DiagnosticInfo>`). Emission
  happens in `RegisterSourceOutput` from that value alone. (Synto's load-bearing discipline — the example
  must model it correctly, since "show how it's done right" is part of the point.)
- **C-6 — Self-contained generated output.** The emitted reader + interceptor reference only BCL
  (`System.Data`, `System.Collections.Generic`) and the consumer's own types — no `Synto.*` runtime
  dependency in the consumer (Synto's injected surface is generator-internal).

## 8. Testing

- **Snapshot (`CSharpGeneratorDriver` + Verify).** The golden `*.g.cs` pins the generated reader +
  interceptor shape for a representative `Person` call. The repo's established pattern; an unexplained
  snapshot change is a finding.
- **Behavioral roundtrip.** The Tests project references `.Generator` as an analyzer, so test code calls
  `ObjectReader.Create(people, "Name", "Age")` directly; the generator runs on the test, and the test reads
  rows back via `DataTable.Load(reader)`, asserting values, column names, field types, and `DBNull` for a
  null member (C-3). Real proof it *works*, not just that it emits.
- **Diagnostics.** A call with an unknown member name asserts `SOR0001` is reported and that the resulting
  reader omits the bad column (C-2).
- **Demo as living proof.** `dotnet run` on `.Demo` produces deterministic console output (used as a smoke
  check, not a snapshot).
- **Green gate (per unit):** `dotnet build -c Debug` (0 errors; analyzer warnings are findings, not gate
  failures) · `dotnet test --no-build -c Debug` (all green; **no `--nologo`** — MTP treats it as unknown
  and runs zero tests) · `dotnet format whitespace --verify-no-changes` (clean; whitespace scope only).

## 9. Key risks / implementation spikes

- **R1 — Generic-call interception (do first).** Validate that a `Create<T>(…)` call where `T` is inferred
  can be intercepted and routed to a type-specialized reader, and settle the interceptor signature/cast
  shape. **Mitigation:** Task 1 is a *walking skeleton* — scaffold all four projects and prove a
  **hand-written** interceptor routes one hard-coded `Person` call to a minimal reader that `DataTable.Load`
  consumes, *before* any Synto templating. If generic interception proves infeasible, fall back options
  (in order): generic interceptor dispatching on `typeof(T)`; or revisit dispatch with the project owner.
- **R2 — `[Template]` expressing list-driven switch arms.** Strategy A needs a template whose holes are
  *N* switch arms computed from the member list. This is the core Synto stress point. **Mitigation:** D3 —
  drop that one hole to raw `SyntaxFactory` (or try `Pattern.Replace`) if the template can't express it,
  and log it. Either outcome is a valid result of the experiment.
- **R3 — Interceptor plumbing in the consumer.** `<InterceptorsNamespaces>` must be set in Demo + Tests;
  document it as required consumer setup (and as friction).

## 10. Scope / non-goals

- **In scope:** the four-project example; `Create<T>(IEnumerable<T>, params string[])`; interceptor
  dispatch; a functionally-correct specialized `IDataReader` (C-3); `SOR0000/0001/0002`; snapshot +
  behavioral + diagnostics tests; a runnable Demo; the friction log (§11).
- **Non-goals (this increment):** FastMember's `TypeAccessor` (get/set by name) — only the `ObjectReader`
  half; a runtime reflection fallback for non-constant calls (C-4 throws instead); set/write or
  `IDataRecord` mutation; nested/flattened member paths (`"Address.City"`); custom column ordering/renaming
  beyond call order; `GetSchemaTable`/`GetData`; packaging as a real NuGet; **any change to Synto itself**
  (phased to a later spec, §11).

## 11. Relationship to Synto improvements (phased)

This example is built against Synto **as-is**. The deliverable beyond the working code is
`docs/superpowers/notes/2026-06-22-objectreader-synto-friction.md` — a **living friction log** appended
throughout the build: every verbose construct, every strategy-A→B drop (R2), every "Synto can't express
this," every bit of non-Synto plumbing (R1/R3), with a concrete "Synto could make this easier by …" note
where one is apparent. That log is the **input to a separate Synto-improvement spec**, written after this
example lands. We do not improve Synto inline (it would muddy the "how painful is it" measurement and
balloon this plan). A genuine hard blocker is logged and worked around; at most one minimal targeted Synto
fix is pulled forward, and only if it blocks the walking skeleton.

## 12. Resolved decisions

- **Concept (resolved).** FastMember-equivalent `ObjectReader` (`IEnumerable<T>` → `IDataReader`) via
  source-gen, not reflection. The `TypeAccessor` half is out (§10).
- **Dispatch (resolved).** FastMember-identical interceptors (D2), not POCO attributes — plus the unknown
  member diagnostic (D5/`SOR0001`).
- **Fidelity (resolved).** Fully functional + runnable (D4, C-3), proven by `DataTable.Load` and a Demo.
- **Synto-change scope (resolved).** Phased — example first against Synto as-is, friction log feeds a later
  Synto spec (§11).
- **Test location (resolved).** Co-located under the example umbrella (D6).
- **Branch/VCS (resolved).** New `experimental/object-reader` bookmark off the `experimental/matching` tip;
  **jj-native**, never main, operator advances the bookmark.
