# issue-implement

Closes the loop for **one** issue: implements issue `<n>`'s ready plan and reconciles `<n>` on GitHub. Like every other `issue-*` skill, it operates on **exactly the `<n>` you pass** ‚Äî never the whole `status:ready` set. (The driver implements many ready issues by *looping* this one-issue stage, not by batching.)

**Read first:** `.claude/rules/github.md` (the Implement comment template, ¬ß work-graph linkage, ¬ß Working-tree hygiene) and `.claude/rules/rigor.md` (the **implement ‚Üí rigor** knob: a plan tagged `> **Rigor:** one-shot` implements in one pass behind the floor; the floor ‚Äî green-gate + final diff-review ‚Äî always runs).

**On tooling failure** (a `gh`/git/tool error, exception, or broken reference ‚Äî *not* the work itself failing, which parks `blocked` per step 6): follow github.md ¬ß Reporting a broken skill (search-first; file/recur a de-duped `issue-flow-bug`).

## Steps

1. **Guard (fail closed).** `gh issue view <n> --json labels,comments`. If `status:ready` is **absent**, refuse: print "Issue #<n> is not status:ready ‚Äî promote its plan first via /issue-respond." and stop.
   - **Unaddressed-comment guard** (github.md ¬ß The unaddressed-comment guard + ¬ß Trust boundary): run `bash .claude/scripts/trust.sh new-human <n>` ‚Äî it returns the trusted, human, unaddressed comments on `<n>` (untrusted/skill comments filtered out mechanically). If it is **non-empty** and its latest comment is **not** a clear go-ahead/ack, do **not** implement ‚Äî `set_status <n> implementing block`, post the `## üõë Needs you ‚Äî unaddressed comment` note, and **stop**. Empty (or a go-ahead/ack) means the discussion is resolved; continue normally.
2. **Resolve `<n>`'s plan slug.** Find `<n>`'s **promoted** plan ‚Äî the file directly under `docs/superpowers/plans/*.md` (top level only, **not** `drafts/`/`completed/`) whose header has `> **Tracking issue:** #<n>` ‚Äî and take its slug (filename minus the `YYYY-MM-DD-` date prefix and `.md`). If `<n>` has **no** top-level plan it isn't actually promoted: refuse ‚Äî print "Issue #<n> has no promoted top-level plan ‚Äî promote it via /issue-respond." and stop (never implement an un-promoted issue).
3. **Claim ‚Üí** `set_status <n> implementing` (single-select; clear `blocked`), marking `<n>` owned before any code lands.
4. **Implement (scoped to `<n>`'s one plan).** `Workflow({ name: 'implement-plan', args: { plan: <the resolved top-level plan path from step 2>, mode: 'merge' } })` ‚Äî `implement-plan` takes exactly **one** plan by path, so no other plan (tracked or untracked) is ever swept. **Adaptive rigor is transparent here:** `implement-plan` reads the plan file's `> **Rigor:**` header itself (rigor.md), so you pass nothing extra ‚Äî a `one-shot` plan implements the whole change in a single pass, a `tdd-per-task` plan (the default) runs the per-task loop. The implement guards hold **in both modes** behind the **floor**: for `tdd-per-task`, a fresh subagent per TASK doing **strict red-green TDD** (failing test first, then minimal code to green) + per-task reviews; for `one-shot`, a single change pass; and **always** the **green-gate** and a **final diff-review** before any push, plus **continuous integration** ‚Äî after every GREEN COMMIT it rebases the worktree branch onto `origin/main`, re-runs the full green-gate, and **fast-forward-pushes the green commit straight to `main`**. It then archives the implemented plan to `completed/`. All code is committed **inside `implement-plan`'s isolated worktree** and ff-pushed; this skill makes **no** commit in the primary checkout and must leave its working tree clean (github.md ¬ß Working-tree hygiene).
5. **Reconcile `<n>`.** Read the single `planResult` the workflow returns. If it reports `<n>`'s plan implemented+archived (`merged === true && reason === 'merged + archived'`):
   - Post the **Implement comment** (github.md template) with the merged commit SHAs (auto-linking).
   - `gh issue close <n> && set-status <n> done` ‚Äî close, then strip any `status:*`/`blocked` label and move the board card to Done (github.md ¬ß Setting state).
6. **Parent rollup (¬ß work-graph).** If `<n>` has a parent and **all** of that parent's sub-issues are now closed, post a brief rollup comment and `gh issue close <parent>`.
7. **On failure.** If the workflow throws/returns `{aborted:true}`, or `<n>`'s plan does not implement, leave `<n>` at `status:implementing` with `blocked` and a failure comment carrying the error ‚Äî **never** close it, and never strand it un-`blocked` (an `implementing`+`blocked` issue is recoverable: a human go-ahead routes it back to retry). Two caveats a manual run should know:
   - **Partial land** (the driver handles this too): if `implement-plan` reports it landed **some** task increments on `main` then stopped, that is **terminal** ‚Äî a bare retry would write a failing test for code that already exists. Recover by trimming the promoted plan to the un-landed tasks (or reverting the landed commits) before retrying.
   - **One-shot upshift** (rigor.md ¬ß Upshift): a `reason` of `one-shot aborted (needs full plan)` or `one-shot floor unresolved (needs full plan)` means the change was bigger/riskier than the `one-shot` tag assumed and **nothing landed** (a one-shot never pushes until the floor passes). Recover by removing the `> **Rigor:** one-shot` line from the plan (retry as `tdd-per-task`) **or** routing the issue back to `plan-queued` for a full plan ‚Äî then clear `blocked`.
8. **Report** whether `<n>` was closed or left `blocked`.
